⚙ Config System config/config_loader.py

Unico punto di lettura e merge della configurazione — global_config + instances + runtime_overrides.

Il config system è il layer che trasforma i file JSON su disco in oggetti tipizzati usati dal bot. Viene riletto ad ogni tick: le modifiche dalla dashboard hanno effetto immediato senza restart. In caso di errore di lettura o campo mancante, i valori di default hardcoded garantiscono che il bot non crashi mai per una configurazione corrotta.

Bootstrap static → dynamic (WU140)

_build_runtime_from_static(global_config, instances) copia tutti i campi configurabili da STATIC a DYNAMIC, escludendo mumu e _note (read-only architetturali). Chiamata da bootstrap_runtime_from_static_if_missing() al boot del bot (main.py) solo se runtime_overrides.json è assente. 

# Utilizzo in main.py (ad ogni tick):
gcfg     = load_global()                          # rilegge global_config.json
overrides = load_overrides(RUNTIME_OVERRIDES_PATH) # rilegge runtime_overrides.json

cfg      = build_instance_cfg(ist_dict, merged, overrides)
ctx      = TaskContext(instance_name=ist["nome"], config=cfg, state=state, log=logger)

Regola STATICO vs DINAMICO (WU140)

STATIC (global_config.json + instances.json): modificabile SOLO da /ui/config/global. Serve da baseline e per il bootstrap.
DYNAMIC (runtime_overrides.json): modificabile SOLO dalla HOME dashboard. Hot-reload: il bot rilegge ad ogni tick, le modifiche hanno effetto immediato senza restart.
I due piani sono indipendenti durante il funzionamento. Il merge avviene in config/config_loader.py::merge_config(gcfg, overrides) con priorità DYNAMIC > STATIC > default. Al primo avvio (override assente), bootstrap_runtime_from_static_if_missing() copia static→dynamic.

Lifespan — background task

All'avvio, lifespan() crea un asyncio.Task che ogni 30s controlla: (a) se è iniziato un nuovo ciclo bot → snapshot immediato cycle predictor; (b) se sono trascorsi 15 min → snapshot ordinario; (c) valuta cicli completati per aggiornare data/predictions/cycle_accuracy.jsonl. Al shutdown il task viene cancellato pulitamente.

Pattern merge endpoint PUT (WU139)

Tutti gli endpoint PUT/PATCH che modificano configurazione Pydantic usano il pattern merge incrementale: parse JSON raw + setattr field-by-field solo per le chiavi esplicitamente presenti nel payload. Non usare mai Pydantic default_factory per i payload in arrivo — i default sovrascriverebbero i campi non inviati (es. acciaio_abilitato tornava False al salvataggio zaino). 

# Pattern corretto endpoint PATCH (WU139):
raw = await request.json()
for k, v in raw.get("zaino", {}).items():
    if hasattr(ov.globali, k):
        setattr(ov.globali, k, v)   # solo campi presenti
_save_ov(ov)                        # scrittura atomica

⚙ Orchestrator core/orchestrator.py

Coordina l'esecuzione periodica dei task su una singola istanza: scheduling, gate HOME, telemetria, last_run tracking.

L'orchestratore registra task con priorita' e tipo di scheduling (periodic/daily), e ad ogni tick() esegue in ordine di priorita' quelli "dovuti". Prima di ogni task esegue un gate HOME (via navigator). Ogni task e' avvolto in try/except per isolare i fallimenti. La persistenza di last_run e' responsabilita' del chiamante.

Schedule types

TipoComportamento
"periodic"Eseguito ogni interval_hours ore dall'ultimo run. Il task definisce task.interval_hours.
"daily"Eseguito una volta per ciclo giornaliero (reset 00:00 UTC, coerente col reset gioco).
"always"Trattato come periodic con interval=0. Il guard interno should_run() decide se eseguire.

Catena di esecuzione per ogni task

1. e_dovuto() → interval/daily scaduto? (scheduling temporale) Se False → skip silenzioso 2. should_run() → flag dashboard abilitato + guard stato persistente Se False → skip con log, last_run NON aggiornato 3. Gate HOME → navigator.vai_in_home() Se fallisce → TaskResult.fail + last_run NON aggiornato → retry al prossimo tick Se ADBUnhealthyError → abort tick + segnale reset istanza 4. task.run(ctx) → esecuzione effettiva 5. last_run → aggiornato SOLO se result.success=True OR result.skipped=True (WU79) Su fail → last_run invariato → retry al prossimo tick

publicenable(task_name: str)Abilita task per nome.
publicdisable(task_name: str)Disabilita task per nome (non eseguito in tick).

🔍 TemplateMatcher shared/template_matcher.py

📦 BaseTask / TaskContext / TaskResult shared/core

Contratto base per ogni task: interfaccia ABC Task, contesto di esecuzione TaskContext, risultato immutabile TaskResult.

Tutti i task del bot implementano l'ABC Task definita in core/task.py. Il pattern e' sincrono per design: nessun async, nessun await. Il contesto TaskContext viene costruito dal launcher e passato immutato a tutti i task del tick.

Esempio — implementazione minimale di un task

from core.task import Task, TaskContext, TaskResult

class MioTask(Task):
    def name(self) -> str:
        return "mio_task"

    def should_run(self, ctx: TaskContext) -> bool:
        # Controlla flag dashboard
        from config.config_loader import task_abilitato
        return task_abilitato("mio_task")

    def run(self, ctx: TaskContext) -> TaskResult:
        ctx.log_msg("[MIO_TASK] avvio")
        screen = ctx.device.screenshot()
        result = ctx.matcher.find_one("pin_target.png", screen.frame)
        if not result.found:
            return TaskResult.fail("target non trovato")
        ctx.device.tap(result.cx, result.cy)
        import time; time.sleep(2.0)  # DELAY UI vincolante
        return TaskResult.ok("completato", taps=1)
